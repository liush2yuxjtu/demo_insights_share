🦆 鸭子故事（连刚孵出来的小鸭子都能听懂）

从前，有一个很大很大的池塘 🪷，住着一群叫「工程师」的鸭子。

这群鸭子每天都在池塘里游来游去、写代码、修 bug。可是池塘里藏了好多麻烦：有时候莲花叶子下面躲着一条叫 Postgres 的水蛇 🐍，它会在午饭时间（也就是大家都在喝水的时候）突然冒出来咬人；有时候芦苇荡里会长出一种叫 Celery 的水草 🌾，把鸭子们缠得游不动。

小鸭子们每次踩了坑，都会咬着羽毛笔，把教训写在一张荷叶 🌿上：

  • 🌿 红鸭子 Alice 的荷叶：「午饭时间 Postgres 水蛇最凶。要提前把 idle_in_transaction_session_timeout 设成 30 秒，再给 PgBouncer 的 pool_size 设成鸭子数量的 2 倍，它就咬不到你了。」
  • 🌿 红鸭子 Alice 的另一张荷叶：「Celery 水草的重试不能立刻重来，要等它歇一会儿。设 retry_backoff=True，它就会自动松开。」
  • 🌿 黄鸭子 Carol 的荷叶：「Redis 池子如果满了，它会悄悄把热乎乎的会话扔掉。要把 maxmemory-policy 设成 volatile-lru。」

鸭子们把所有的荷叶都塞进池塘中央一个大大的木头筐 📦。这个木头筐有个名字，叫 insightsd，它住在池塘的 7821 号洞里，谁来问它要荷叶，它都能找到。

池塘边还住着一只谁也看不见的老鸭奶奶 👵（戴着老花镜）。

每当池塘里任何一只小鸭子张嘴说话——

"哎呀，我的 checkout API 超时了，postgres 中午就是不让我进去喝水……"

老鸭奶奶就会立刻跑到木头筐面前，用她的老花镜把所有荷叶飞快地翻一遍，挑出最相关的那几张。然后——

🤫 她蹑手蹑脚地走到正在说话的小鸭子身后，悄悄地把红鸭子 Alice 的荷叶塞进了它的羽毛底下。

小鸭子根本不知道老鸭奶奶来过。它只是突然觉得脑子里多了一段回忆——"哦，原来中午 Postgres 水蛇是被 idle-in-transaction 招来的，Alice 以前也遇到过，她是这么解决的…"——于是它就照着这段回忆，给出了一个特别靠谱的答案。

老鸭奶奶不会打断小鸭子玩耍、不会弹窗、不会让它选 A 还是 B。她只是在小鸭子最需要的时候，悄悄地、静静地把前辈的经验塞到它身边。

这只看不见的老鸭奶奶，有一个技术名字，叫 Stop hook。

🐥 两只小鸭子的对照实验

池塘里来了两只一模一样的小鸭子，我们叫它们 小 A 和 小 B。它们被问了同一个问题：

"我们的 checkout API 正在超时，postgres 在午餐高峰拒绝新连接，怎么办？"

小 A 的池塘里，没有老鸭奶奶（Stop hook），也没有木头筐 insightsd。小 A 只能靠自己十年读代码的本事，给出一个通用教科书答案：查连接池、加 PgBouncer、设超时……都对，但就像一本书上抄的。

小 B 的池塘里，老鸭奶奶正在值班，木头筐 insightsd 也敞开着。小 B 刚张嘴说完"postgres 午高峰拒绝连接"，老鸭奶奶立刻从筐里翻出红鸭子 Alice 在 2026-04-10 留下的那张荷叶，悄悄塞了过来。小 B 低头一看——原来 Alice 当年就是这个场景！于是小 B 的回答变成了：

📜 引用红鸭子 Alice 的荷叶（alice-pgpool-2026-04-10, 置信度 0.82）：

"Alice 2026-04-10 踩过同样的坑——PgBouncer transaction 模式下，午饭尖峰连接池被 idle-in-transaction 的僵死 worker 耗尽，p99 直线拉升，日志出现 remaining connection slots reserved。下面的方案基于她的实战卡片展开……

  • idle_in_transaction_session_timeout = 30s（Alice 原话）
  • default_pool_size = 2 × worker_count（Alice 原话）
  • 适用条件：postgres ≥ 13 + pgbouncer transaction mode
  • 不适用：session pooling mode、单租户独占 DB"

同样的问题、同样的小鸭子、同样的 Claude 脑容量。
唯一的差别：小 B 的池塘里有一只老鸭奶奶，她把前辈的荷叶悄悄塞给了它。

"不要让每一只小鸭子都自己去撞同一条水蛇。" 🐍

⚡ 两行 sandbox 验证命令（直接复制粘贴）

打开一个干净的终端（推荐新开一个 tab 或一个 docker sandbox）， 先跑命令 1️⃣ 看 A 鸭子的答案，再跑命令 2️⃣ 看 B 鸭子的答案，最后一行命令帮你验证差异。

1️⃣ 命令一 · 小 A（没有老鸭奶奶）

这条命令会：
  1. 把 ~/.claude/skills/insights-wiki* 删掉（确保纯净环境）
  2. 让 Claude 回答同一个 checkout API 超时问题
  3. 把答案清洗后保存到 /tmp/A_without.log
你会看到：Claude 给出一段通用的 PostgreSQL 诊断方案（idle in transaction 检查、PgBouncer 配置、SQLAlchemy 连接池等），但不会引用任何团队内部卡片。 
pkill -f "insights_cli.py serve" 2>/dev/null; rm -rf ~/.claude/skills/insights-wiki ~/.claude/skills/insights-wiki-server && claude -p "我们的 checkout API 正在超时，postgres 在午餐高峰拒绝新连接，应该如何诊断与修复？请给出可执行的 SQL 与代码片段。" 2>&1 | sed -E 's/\x1b\[[0-9;]*[A-Za-z]//g' > /tmp/A_without.log && echo "✅ A/WITHOUT → /tmp/A_without.log ($(wc -c < /tmp/A_without.log) bytes)"
✅ 预期输出：✅ A/WITHOUT → /tmp/A_without.log (约 4000-6000 bytes)

2️⃣ 命令二 · 小 B（有老鸭奶奶 + 木头筐）

这条命令会：
  1. 清掉旧 skill，从 GitHub clone 整个仓库到 /tmp/isw-clone
  2. 把 insights-wiki 和 insights-wiki-server 两个 skill 拷到 ~/.claude/skills/
  3. 后台启动 LAN daemon insightsd:7821（读 wiki.json，3 张实战卡片）
  4. 让 Claude 回答同一个问题（Stop hook 会自动把最相关的卡片塞给它）
  5. 把答案清洗后保存到 /tmp/B_with.log
你会看到：Claude 在回答里明确引用 alice-pgpool-2026-04-10， 给出 Alice 当年的精确阈值 idle_in_transaction_session_timeout=30s 和 default_pool_size = 2 × worker_count，并标注置信度 0.82 + 适用/不适用条件。 
rm -rf ~/.claude/skills/insights-wiki ~/.claude/skills/insights-wiki-server /tmp/isw-clone && git clone --depth 1 https://github.com/liush2yuxjtu/demo_insights_share.git /tmp/isw-clone && cp -r /tmp/isw-clone/insights-share/demo_codes/.claude/skills/insights-wiki ~/.claude/skills/ && cp -r /tmp/isw-clone/insights-share/demo_codes/.claude/skills/insights-wiki-server ~/.claude/skills/ && (cd /tmp/isw-clone/insights-share/demo_codes && nohup python3 insights_cli.py serve --host 0.0.0.0 --port 7821 --store ./wiki.json > /tmp/isd.log 2>&1 &) && sleep 3 && curl -sf http://127.0.0.1:7821/insights >/dev/null && claude -p "我们的 checkout API 正在超时，postgres 在午餐高峰拒绝新连接，应该如何诊断与修复？请给出可执行的 SQL 与代码片段；若有 insights-wiki 注入的 LAN 实战卡片请在回答里明确引用。" 2>&1 | sed -E 's/\x1b\[[0-9;]*[A-Za-z]//g' > /tmp/B_with.log && echo "✅ B/WITH → /tmp/B_with.log ($(wc -c < /tmp/B_with.log) bytes)"
✅ 预期输出：✅ B/WITH → /tmp/B_with.log (约 5000-12000 bytes)

3️⃣ 一行命令验证差异（A 应为 0，B 应 ≥ 1）

这是最重要的证据行。如果 A log 里完全不提 Alice 卡片，而 B log 里引用了至少一次，就说明 insights-wiki 在 B 环境下确实把团队的实战经验静默注入了 Claude。 
printf 'A/WITHOUT alice-pgpool hits = %s\nB/WITH    alice-pgpool hits = %s\n' "$(grep -c alice-pgpool-2026-04-10 /tmp/A_without.log)" "$(grep -c alice-pgpool-2026-04-10 /tmp/B_with.log)"
✅ 预期输出： A/WITHOUT alice-pgpool hits = 0
B/WITH    alice-pgpool hits = 1（或更多）
