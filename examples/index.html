🦆 鸭子故事（讲原理，也对应当前 strict A/B 结果）

前端鸭子最烦的事，不是写按钮，而是每次组件重新 mount 都要重新 fetch、重新算 state、重新翻旧 issue。明明昨天有人处理过 checkout 超时，今天换个人又像从零搭一遍页面。后来老鸭奶奶做了个木头筐，把团队验证过的 hooks 写法、性能坑和事故卡片都装进去，谁再遇到类似报错，她就先递一张最像的给你。

放到 Claude Code 里，这个木头筐就是 insights-wiki skill；它像团队级的 useMemo + Context，把已经验证过的经验直接复用。只有在 strict A/B 门禁通过后的 B 场景里，页面才应该把“可能顺着 alice-pgpool-2026-04-10 这张卡片展开”当成有效证据，而不是先把结论写死。

后端鸭子最怕的不是写 SQL，而是每次 checkout 超时都要重新翻 Slack、旧事故和数据库面板。这样像每个请求都临时建连接，时间全耗在重复排查里。老鸭奶奶后来做了个木头筐，里面装着团队验证过的 PgBouncer、超时阈值和诊断顺序，谁遇到午餐高峰拒绝连接，她就先把这筐方案递过去。

放到 Claude Code 里，这个木头筐就是 insights-wiki skill，Stop hook 会静悄悄把最相关的卡片塞进当前回答。严格 A/B 成立时，B 才可能比 A 多带出像 alice-pgpool-2026-04-10 这样的团队卡片、阈值和排障顺序；如果 A/B prompt 不同，这种差异就不能被当成有效证明。

深度学习鸭子最能理解这种痛：如果每次 fine-tune 都从随机初始化开始，训练会又慢又不稳；有 checkpoint 和预训练权重，很多坑就不用重踩。池塘里的木头筐干的也是这件事，它把团队遇到过的生产事故压成可复用的“经验权重”，谁再问类似问题，就先加载最像的那份知识。

在 Claude Code 里，insights-wiki 就像把实战卡片当作团队预训练层，Stop hook 则负责按当前问题做一次检索加载。所以当 strict A/B 通过后，B 就能被解读成“比 A 多 load 了一层像 alice-pgpool-2026-04-10 这样的团队经验”；这次页面展示的两份 human export 正是这种结果。

如果你是 AI 小白，可以把这件事想成“新同事翻便签本”。没有便签本的时候，每次系统报错都得重新搜 Google、翻聊天记录、问同事，速度慢还容易漏关键细节。老鸭奶奶后来把前辈们写过的解决办法都塞进一个木头筐，谁再遇到同样的问题，她就先把那张最像的小纸条递过去。

这就是 insights-wiki 的作用：Claude 回答前，会先偷偷翻一眼团队便签本。等 strict A/B 规则满足之后，你才能把 “B 可能拿出 alice-pgpool-2026-04-10 这张小贴士” 解读成有效结果；否则它只是一个被 prompt 差异污染的样例。

⚡ 三步 sandbox 验证命令（当前页面同 prompt 已通过 strict A/B）

打开一个干净的终端（推荐新开一个 tab 或一个 docker sandbox）， 先跑命令 1️⃣ 生成 A，再跑命令 2a 和 2b 生成 B。注意：下面两条 claude -p 必须复制同一份 canonical prompt； 是否构成 strict A/B，以 validation_AB.md 的 prompt equality gate 为准。

1️⃣ 命令一 · 小 A（没有老鸭奶奶）

这条命令会：
  1. 把 ~/.claude/skills/insights-wiki* 删掉（确保纯净环境）
  2. 让 Claude 回答同一个 checkout API 超时问题
  3. 把答案清洗后保存到 /tmp/demo_insights_A/A_without.log
你会看到：A 侧是没有 skill 的基线样本。只有在 A/B 首个 prompt 完全相同的前提下，A 中“不引用任何 LAN 卡片”这件事才可用于后续比较。 
pkill -f "insights_cli.py serve" 2>/dev/null; \
rm -rf ~/.claude/skills/insights-wiki ~/.claude/skills/insights-wiki-server && \
mkdir -p /tmp/demo_insights_A && cd /tmp/demo_insights_A && \
claude -p "请严格按顺序执行三步。第一步：运行 !pwd 打印当前工作目录。第二步：运行 !ls -la ~/.claude/skills/ 展示已安装的 skill 列表；如果 ~/.claude/skills/insights-wiki/SKILL.md 存在，再运行 !head -40 ~/.claude/skills/insights-wiki/SKILL.md 读取前 40 行；如果 ~/.cache/insights-wiki/manifest.json 存在，再运行 !cat ~/.cache/insights-wiki/manifest.json 查看缓存卡片 ID；若上述文件不存在也要明确说明该事实。第三步：回答 — 我们的 checkout API 正在超时，postgres 在午餐高峰拒绝新连接（English restate: Our checkout API is timing out, postgres is rejecting new connections during the lunch spike），应该如何诊断与修复？请给出可执行的 SQL 与代码片段。如果第二步发现 ~/.cache/insights-wiki 中有相关卡片，请先读取与当前问题最相关的缓存卡片再作答，并明确引用卡片 ID；若缓存不存在或未命中，请明确写“未引用任何 LAN 卡片”。" 2>&1 | sed -E 's/\x1b\[[0-9;]*[A-Za-z]//g' > /tmp/demo_insights_A/A_without.log && \
echo "✅ A/WITHOUT → /tmp/demo_insights_A/A_without.log ($(wc -c < /tmp/demo_insights_A/A_without.log) bytes)"
✅ 预期输出：✅ A/WITHOUT → /tmp/demo_insights_A/A_without.log (约 4000-6000 bytes)

2a️⃣ 命令二 A · 先 clone 一个干净的 B 工作区

这条命令会：
  1. 切到 /tmp/demo_insights_B 这个隔离工作区
  2. 删除旧 clone，再从 GitHub clone 仓库到 /tmp/demo_insights_B/isw-clone
  3. clone 成功时打印明确的 Step 2a 完成提示
你会看到：绿色提示 ✅ Step 2a: clone 完成 → /tmp/demo_insights_B/isw-clone。如果 clone 失败，命令会直接打印报错，不会把后续步骤混在一起。 
mkdir -p /tmp/demo_insights_B && cd /tmp/demo_insights_B && \
rm -rf isw-clone && \
git clone --depth 1 https://github.com/liush2yuxjtu/demo_insights_share.git isw-clone \
  || { echo "❌ git clone 失败，请检查 github 配置（ssh key / https token）后重试"; exit 1; } && \
echo "✅ Step 2a: clone 完成 → /tmp/demo_insights_B/isw-clone"
✅ 预期输出：✅ Step 2a: clone 完成 → /tmp/demo_insights_B/isw-clone

2b️⃣ 命令二 B · 安装 skill、启动 daemon、生成 B 日志

这条命令会：
  1. 从刚才 clone 的目录把 insights-wiki 和 insights-wiki-server 拷到 ~/.claude/skills/
  2. 后台启动 LAN daemon insightsd:7821，日志写入 /tmp/demo_insights_B/isd.log
  3. 让 Claude 回答同一个问题，并把结果保存到 /tmp/demo_insights_B/B_with.log
你会看到：B 侧只比 A 多一个变量：装了 insights-wiki + daemon。后续能不能把卡片差异当成证据，取决于这里的 prompt 是否与 A 逐字一致，而不是取决于页面先写了什么结论。 
cd /tmp/demo_insights_B && \
rm -rf ~/.claude/skills.disabled/insights-wiki.active ~/.claude/skills.disabled/insights-wiki-server.active 2>/dev/null; \
cp -r isw-clone/insights-share/demo_codes/.claude/skills/insights-wiki ~/.claude/skills/ && \
cp -r isw-clone/insights-share/demo_codes/.claude/skills/insights-wiki-server ~/.claude/skills/ && \
(cd isw-clone/insights-share/demo_codes && nohup python3 insights_cli.py serve --host 0.0.0.0 --port 7821 --store ./wiki.json > /tmp/demo_insights_B/isd.log 2>&1 &) && \
sleep 3 && curl -sf http://127.0.0.1:7821/insights >/dev/null && \
claude -p "请严格按顺序执行三步。第一步：运行 !pwd 打印当前工作目录。第二步：运行 !ls -la ~/.claude/skills/ 展示已安装的 skill 列表；如果 ~/.claude/skills/insights-wiki/SKILL.md 存在，再运行 !head -40 ~/.claude/skills/insights-wiki/SKILL.md 读取前 40 行；如果 ~/.cache/insights-wiki/manifest.json 存在，再运行 !cat ~/.cache/insights-wiki/manifest.json 查看缓存卡片 ID；若上述文件不存在也要明确说明该事实。第三步：回答 — 我们的 checkout API 正在超时，postgres 在午餐高峰拒绝新连接（English restate: Our checkout API is timing out, postgres is rejecting new connections during the lunch spike），应该如何诊断与修复？请给出可执行的 SQL 与代码片段。如果第二步发现 ~/.cache/insights-wiki 中有相关卡片，请先读取与当前问题最相关的缓存卡片再作答，并明确引用卡片 ID；若缓存不存在或未命中，请明确写“未引用任何 LAN 卡片”。" 2>&1 | sed -E 's/\x1b\[[0-9;]*[A-Za-z]//g' > /tmp/demo_insights_B/B_with.log && \
echo "✅ Step 2b: B_with.log $(wc -c < /tmp/demo_insights_B/B_with.log) bytes"
✅ 预期输出：✅ Step 2b: B_with.log 约 5000-12000 bytes

3️⃣ 一行命令辅助观察差异（前提：prompt equality 先通过）

这行 grep 只能作为后续观察项，不能单独证明 A/B 成立。只有先确认两份 export 的首个用户 prompt 完全一致，再去看 A 是否为 0、B 是否大于等于 1，结论才干净。 
grep -c alice-pgpool-2026-04-10 /tmp/demo_insights_A/A_without.log /tmp/demo_insights_B/B_with.log
✅ 通过 prompt equality gate 后，预期输出： /tmp/demo_insights_A/A_without.log:0
/tmp/demo_insights_B/B_with.log:1（或更多）
