# Checkout API 超时 & PostgreSQL 连接拒绝诊断

## 一、立即诊断（现在执行）

### 1.1 PostgreSQL 连接状态

```sql
-- 当前活跃连接数 vs 最大允许连接数
SELECT 
  count(*) AS active_connections,
  pg_setting('max_connections')::int AS max_connections,
  count(*) * 100.0 / pg_setting('max_connections')::int AS usage_pct
FROM pg_stat_activity
WHERE state IS NOT NULL;

-- 等待连接的进程（连接堆积）
SELECT pid, usename, pg_blocking_pids(pid) AS blocked_by, wait_event_type, wait_event, query_start
FROM pg_stat_activity
WHERE wait_event_type = 'Lock' OR wait_event_type = 'Client'
ORDER BY query_start;
```

### 1.2 慢查询诊断

```sql
-- 找出最慢的查询（可能是 checkout 阻塞原因）
SELECT pid, now() - query_start AS duration, query, state
FROM pg_stat_activity
WHERE state != 'idle' AND now() - query_start > interval '5 seconds'
ORDER BY duration DESC;

-- 表锁等待（外键未索引会导致写入阻塞）
SELECT blocked_locks.pid AS blocked_pid, blocking_locks.pid AS blocking_pid
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_locks blocking_locks 
  ON blocking_locks.lockable_object = blocked_locks.lockable_object
WHERE NOT blocked_locks.granted;
```

### 1.3 连接来源分析

```sql
-- 各应用的连接数（判断是否池化客户端配置有误）
SELECT application_name, count(*) 
FROM pg_stat_activity 
GROUP BY application_name 
ORDER BY count(*) DESC;
```

## 二、常见根因与修复

### 根因 A：PgBouncer / 连接池配置错误

**症状**：连接数没到 max，但应用侧频繁连接拒绝

```ini
; pgbouncer.ini 检查项
pool_mode = transaction          ; 必须是 transaction，不是 session
max_client_conn = 1000
default_pool_size = 20           ; 计算公式：max_connections / 应用数
```

**错误修复**（来自 bob-pgpool-bad-2026-04-12 的教训）：
> 盲目增大 `default_pool_size` 会导致数据库更卡 — PostgreSQL 是进程模型，每个连接都消耗内存。

```python
# 计算正确的 pool_size
# 公式：default_pool_size = (max_connections - admin_connections) / 应用副本数

# 假设 max_connections = 100, 应用有 4 个 pod
pool_size = (100 - 3) // 4  # = 24，不是越大越好
```

### 根因 B：长事务阻塞 checkout

**症状**：checkout 偶尔超时，但 DB 连接数不高

```python
# 确保 checkout 操作使用短事务 + 索引
import asyncpg

async def checkout_items(conn, user_id: int, item_ids: list[int]):
    async with conn.transaction():
        # 添加 NOWAIT 防止无限等待锁
        async with conn.cursor(
            "SELECT * FROM inventory WHERE id = ANY($1) FOR UPDATE NOWAIT"
        ) as cur:
            await cur.fetch(item_ids)
        # ... 执行业务逻辑，超时自动回滚
```

### 根因 C：缺少连接池健康检查

```python
# FastAPI + asyncpg 连接管理
from contextlib import asynccontextmanager
import asyncpg

@asynccontextmanager
async def get_db_pool():
    pool = await asyncpg.create_pool(
        dsn="postgresql://user:pass@pg:5432/checkout",
        min_size=5,
        max_size=20,          # 不要超过 pgbouncer default_pool_size
        command_timeout=5.0,  # 防止慢查询占满池
        timeout=3.0,          # 连接获取超时
    )
    try:
        yield pool
    finally:
        await pool.close()

# 请求级超时保护
async def checkout_with_timeout(pool, user_id, items, timeout=2.0):
    async with asyncio.timeout(timeout):
        async with pool.acquire() as conn:
            return await do_checkout(conn, user_id, items)
```

## 三、峰值前预防

```sql
-- 设置连接超时警告阈值（提前发邮件）
CREATE OR REPLACE FUNCTION check_connections()
RETURNS void AS $$
DECLARE
  cnt int;
BEGIN
  SELECT count(*) INTO cnt FROM pg_stat_activity;
  IF cnt > (pg_setting('max_connections')::int * 0.8) THEN
    PERFORM pg_notify('high_connections', json_build_object('count', cnt)::text);
  END IF;
END;
$$ LANGUAGE plpgsql;

-- 每分钟检查（可在 pg_cron 或外部监控调用）
SELECT check_connections();
```

## 诊断 SQL

```sql
-- 1. 当前连接数快照
SELECT state, count(*), max(now() - state_change) as max_duration
FROM pg_stat_activity 
WHERE datname = current_database()
GROUP BY state;

-- 2. 找出最慢的查询（很可能在阻塞）
SELECT pid, now() - query_start as duration, state, query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC
LIMIT 10;

-- 3. 检查是否真的有锁等待
SELECT blocked_locks.pid AS blocked_pid, blocking_locks.pid AS blocking_pid
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.granted
WHERE NOT blocked_locks.granted;

-- 4. 连接来源统计（判断是不是连接池配置太小）
```

## 立即修复

**1. 临时扩容连接数**
```sql
-- 看当前 max_connections
SHOW max_connitions;

-- 临时提高（需要 superuser）
ALTER SYSTEM SET max_connections = 500;
SELECT pg_reload_conf();
-- 或重启生效
```

**2. 检查并修复耗尽连接的慢查询**
```sql
-- 杀掉卡住超过 5 分钟的非 idle 连接
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state NOT IN ('idle', 'idle in transaction')
  AND now() - state_change > interval '5 minutes'
  AND pid <> pg_backend_pid();
```

## 代码层修复

**连接池配置（假设用 pgBouncer 或应用层池）**

```python
# 如果是应用层连接池（Python/Node/Java 等）
# 典型配置问题：

import psycopg2
from psycopg2 import pool

# 原来配置（太小）
conn_pool = psycopg2.pool.SimpleConnectionPool(
    minconn=1,
    maxconn=10,      # ← 午餐高峰 10 个根本不够
    ...
)

# 调大并添加超时
conn_pool = psycopg2.pool.ThreadedConnectionPool(
    minconn=5,
    maxconn=50,      # 根据：(平均查询时间 × QPS) / 服务器核心数
    max_connections=100,  # postgres 侧也要配合调高
    command_timeout=30,   # ← 关键：单个查询超时
    connection_timeout=10, # ← 关键：获取连接超时
)
```

**Node.js (pg 库)**
```javascript

const pool = new Pool({ max: 10 });

// 调大 + 超时
const pool = new Pool({
  max: 50,
  connectionTimeoutMillis: 5000,   // 获取连接超时
  statement_timeout: 30000,          // 查询超时
  idleTimeoutMillis: 30000,
});
```

## 根本原因排查

```sql
-- 如果上面的查询发现大量 "idle in transaction"，是代码事务泄露
SELECT pid, now() - query_start as duration, state, query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND now() - state_change > interval '1 minute';
```

典型泄露代码（Python）：
```python
# 错误写法 - 事务没提交也没回滚
def checkout():
    conn = get_conn()  # 用完了没还
    # 如果这里抛异常，conn 永远不会还到池里
    if complicated_logic():
        raise Exception("oops")
    commit(conn)
    return result
```

```python
# 正确写法 - 保证还池
def checkout():
    conn = None
    try:
        conn = get_conn()
        result = do_work(conn)
        commit(conn)
        return result
    finally:
        if conn:
            conn.close()  # 或 pool.putconn(conn)
```

## 预防措施

```sql
-- 设置连接超时警告（pg 13+）
CREATE EXTENSION IF NOT EXISTS pg_stat_kcache();

-- 在 pg_bouncer.ini 里配置
[pgbouncer]
max_db_connections=100
default_pool_size=25
query_timeout=30
query_wait_timeout=30
```

**监控报警阈值（建议）**
- 连接数 > max_connections 的 80% → 报警
- 单查询超过 5 秒 → 报警
- `idle in transaction` 超过 1 分钟 → 报警 + 自动杀

---
