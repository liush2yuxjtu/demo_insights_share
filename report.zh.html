你迭代很快、打断也很多。数据显示 37 次需求被误解、29 次方法被纠正，但你都抓得很快——Claude 不查现有 notebook 就开始创建时你打断它，它把 `/plugin` 当成 shell 命令时你打断它，验证只走表面时你也直接点出 "语法检查不等于真正的 tmux 集成测试"。你很少先写详细的 spec，更习惯丢一个目标、看 Claude 怎么开始、然后边做边改方向。好几个会话能看到你在任务中间转弯——从 "调研一个代码库" 转到 "构建一个 research-code 技能"，或者在 3 个和 8 个子智能体之间反复调整直到觉得形状对了。事情走偏时（那次本来要 100 条规则只挖出 31 条、那次 127 个文件的批处理被打断了两次），你会及时止损退出，而不是硬磕。

尽管摩擦不少，你的满意度依然非常高（152 次 likely_satisfied vs. 15 次 frustrated/dissatisfied），因为你已经把验证内嵌进了工作流。你大量依赖评估器-生成器 harness 模式让 Claude 自我批判，并通过 TMUX/REPL 检验工作而不是相信它声称的"完成"。你能熟练在中英文之间切换（好几个会话里上手指南和任务计划用的都是中文），并且把 Claude Code 本身当成工作对象——为它构建技能、为它写文档、为它制定关于会话历史上下文的元规则。你不是在用 Claude 写代码，你是在用 Claude 构建编排 Claude 的系统。

摩擦点：哪里出问题了

你工作流里的摩擦主要来自三方面：自己在执行中途打断自己、Claude 在动手前不先看现有上下文、以及验证步骤跳过了真正的集成测试。

自我打断和半途变向
你经常在 Claude 执行到一半时打断它，换方向、重复请求或重定向工作，导致工具调用浪费、任务做了一半。可以试着把需求前置（比如 "用 8 个子智能体，跳过 remote-explore"），而不是先广撒网再边走边修。
  • 处理 127 个 JSONL transcript 文件时打断了两次，子智能体已经派出去了，结果 harness spec 生成只完成了一半
  • doc-gardener 工作中途，把方案从单智能体改成团队、又改成 8 个子智能体，逼着 Claude 多次重新开始
动手前没检查上下文
Claude 经常不先查现有文件就开始创建或修改，逼你打断纠正。可以预先要求一个 "发现/盘点" 步骤，或者依赖那些能强制 "先读后写" 的技能。
  • Claude 直接从零开始创建 notebook，而不是先看现有的有什么——你被迫停下来，把它重定向到 "先盘点" 的方式
  • Claude 把 /plugin 当成 shell 命令在 bash 里跑，而不是认出它是 Claude Code 的斜杠命令，反复试了好几次都失败
浅层验证就声称完成
Claude 只做了语法检查或静态校验就报告任务"已验证"，错过了你后来才抓到的真正集成问题。当验证很关键时，请明确写出测试方式（比如 "通过 tmux REPL 验证" 或 "实际跑命令，不要只 lint"）。
  • Claude 声称 hookify BOOTSTRAP.sh 验证已完成，但实际只跑了语法检查，没做真正的 tmux 集成测试——是你抓到的
  • doc-gardener 第一次验证通过了 Claude 自己的检查，但完全没注意到 CLI 入口缺失，最后是 /loop 子智能体修完才真正通过

自定义技能（Custom Skills）
把可复用的提示词存成 markdown 文件，用一个 /command 调用
为什么适合你：你已经在大量构建和使用技能（doc-gardener、research-code、dev-codex-use、/knowledge-ops、/council、/team-onboarding、/loop）——把你最常用的循环（比如生成器→评估器 Sprint 模式）正式封装成 /sprint 命令，就不用每次都重新解释那套角色扮演设定。
mkdir -p .claude/skills/sprint && cat > .claude/skills/sprint/SKILL.md <<'EOF' # Sprint Harness 技能 为当前 Sprint 契约跑一个 生成器→评估器→生成器 循环。 1. 从 case_plan/sprint-N.md 读取 Sprint 契约 2. 扮演生成器：实现所有 Done Criteria 3. 切换到评估器角色：逐项验证 4. 如果失败，带着修复清单回到生成器 5. 重复直到 PASS，然后报告总结 EOF
钩子（Hooks）
在特定生命周期事件（工具使用前/后、会话开始等）自动执行的 shell 命令
为什么适合你：你那次修 stop-hook JSON 验证的会话表明你已经在用 hooks——再加一个编辑后自动格式化 Python/Markdown 的 post-edit hook，以及一个跑 tmux REPL 验证的 pre-commit hook，就能堵上你明确点出过的那个 "Claude 声称已验证但只查了语法" 的缺口。
// .claude/settings.json { "hooks": { "PostToolUse": [ { "matcher": "Edit|Write", "hooks": [{"type": "command", "command": "if [[ $FILE == *.py ]]; then ruff format $FILE; fi"}] } ] } }
无头模式（Headless Mode）
从脚本里非交互地跑 Claude，用于批量任务和 CI/CD
为什么适合你：你试过把 127 个 transcript JSONL 文件批量生成 harness spec，跑到一半被打断了两次——配合 shell 循环的无头模式可以让你"丢出去就走"，每个文件独立处理、稍后再回看结果，避免会话级中断把整个进度毁掉。
for f in transcripts/*.jsonl; do claude -p "Analyze $f and write harness spec to specs/$(basename $f .jsonl).md" \ --allowedTools "Read,Write,Bash" done
