S — 现状（Situation）

claude /insights 本身有价值，但三个核心问题让它无法团队化
问题 1
生成耗时过长
每次冷启动都要重新跑完整 insight 生成流程，约需 60+ 秒。
问题 2
insights 仅对个人有效
生成出来的 insight 存在个人 profile 里，其他团队成员完全看不到。
问题 3
直接复用导致次优表现
直接把别人的 insight 套用到自己的上下文，结果可能比没有更差。
以上三点是初版痛点
这就是 MVP 的起点
后续在 0414 两次更新中加入了更多设计约束（见下）。
📌 update 0414 09:29 — 新增 4 条设计约束
  • 4
    SILENT-IN-BACKGROUND：上传 / 下载 insights 必须在后台静默完成，用户无感知。静默后台
  • 5
    强制下载：只要用户加载了我们的 insights 工具，就自动把最新 insights 推送到本地，用户无需主动操作。强制推送
  • 6
    管理员 CRUD：管理员可以对 wiki-insights 做增删改查和审核。Demo 阶段所有人都是管理员。Admin CRUD
  • 7
    skill 形式交付：wiki-insights 目前以 Claude Code skill 的形式存在和被调用。Skill
🎯

T — 任务（Task）

在局域网内制作一个 insights-share 功能 demo，CLI 优先
  • 1
    生成一条 insight：展示 insight 从产生到入库的完整路径。
  • 2
    公司 wiki / 渐进式披露系统：有一个中央存储，分类管理所有团队 insights。
  • 3
    遇到新问题先检索：每次遇到问题时，优先从公司 wiki 语义检索相关 insights，而非冷启动生成。
  • 4
    仅热加载 wiki-insights：不全量加载，只取最相关的那一张卡热加载，保持速度。
  • 5
    用户无感知比对验证：在后台快速验证 insight 是否适用于当前上下文，不打断用户。
  • 6
    携带高层次 insights 继续工作：把验证通过的 insight 注入当前工作上下文，让 agent 带着"前人经验"继续。
设计约束：demo 必须基于 CLI
AGENTS 更容易通过 CLI 使用；如需 REPL，应提供用于启动守护进程或服务器的 CLI 命令。 CLI-first

A — 行动（Action）

如何执行这份提案
定位
面向 PM，而非开发者
演示的语言、界面、交互流程，以 PM 能直接看懂为标准。不要堆技术术语。
产品形态
MVP · 即点即用
展示一个可运行的最小产品，而非原型图或文档。要能当场跑出来。
执行策略
请立即开始执行
先执行再门控——先产出可检查的结果，再请用户确认，而不是反复确认再动手。
目标受众
也展示给其他 agent 开发者
