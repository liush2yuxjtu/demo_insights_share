验证任务 1 · 触发率

用 20 个用例验证 hook 的触发精度，12/8 训练测试划分

验证体系的第一步：量化系统"知道什么时候该出手"的能力。

✅ 应触发（10 个用例）
  • 提到 PostgreSQL 连接池
  • 提到 Redis LRU / session eviction
  • 提到 Celery retry storm
  • 类似已知故障模式的描述
  • …共 10 个正样本
🚫 不应触发（10 个用例）
  • 普通 Python 语法问题
  • 无关领域（前端 CSS 等）
  • 闲聊 / 问候语
  • 已在 wiki 中标记为低置信度的条目
  • …共 10 个负样本
训练集（12 个）
  • 用于调整触发关键词权重
  • 优化 INDEX.md 触发条件描述
测试集（8 个）
  • 保持盲测，不参与调参
  • 触发率必须达标才视为通过
迭代原则：持续优化触发率，直到达标。"达标"由每次 demo 前的验证报告决定，而非一次性定义。

验证任务 2 · 优化触发效果

用 Alice & Bob 案例演示完整触发链路，并确认 SILENT_AND_JUST_RUN 机制

这是整个 demo 的核心故事：Bob 不主动求助，系统自动识别并提前送上 Alice 的经验。

S · 现状
来自 Alice 和 Bob 案例（pm_walkthrough.md）：Bob 上周 Alice 已经解了 PostgreSQL 连接池被午高峰打爆的故障，insight 却躺在 Alice 个人 profile 里。
T · 任务
Bob 打开 Claude Code，输入与 PostgreSQL 相关的内容时，不是主动求助，但可能遇到同样的陷阱。
A · 行动
Claude 发出最后一张图片后，Claude Code 触发一个 haiku-agent 对 insights-wiki 进行语义搜索并汇报最终结果。haiku-agent
R · 结果
Claude Code 审查相关问题，Bob 不需要知道背后发生了什么，已经自动携带 Alice 的经验继续工作。

触发机制选型

  • 选定
    SILENT_AND_JUST_RUN：静默直接执行，用户无感知，系统自动在后台完成 haiku-agent 调用和结果注入。静默
  • 弃用
    ASK_USER_APPROVAL（可选项，本设计不采用）：请求用户确认。保留为占位符，但强制默认值为 SILENT_AND_JUST_RUN，不打断用户流。

    验证任务 3 · 更新 Wiki

    5 种 wiki 更新操作，确保知识库不腐化
    • 合
      合并：同一条目存在旧版和新版时，合并为一条，不留冗余。 Merge
    • 标
      标记未触发：绝不遗忘——对于长期未被触发的条目，标记状态而非删除，保留历史。
    • 新
      新增：遇到新的已解问题模式，立即创建新 wiki 条目。
    • 删
      删除：条目被证明无效或完全过时时，可删除。（删前需标记并留存归档版本。）
    • 研
      调研：对存疑的条目标记为"待调研"状态，等待验证后再决定去留。
    • 编
      编辑：对现有条目做内容修正或补充，更新 fix 字段、增加 example，或调整触发条件。
    这 5 种操作覆盖了 wiki 的完整生命周期，让知识库既能增长、也能自我清洁，不因时间流逝而变成垃圾堆。
    🏗️

    验证任务 4 · Wiki 结构

    强制 4 层结构：wiki_type → INDEX.md → item.md → 原始日志

    Wiki 类型（wiki_type）

    general_python frontend_python database infra_cache infra_queue …（按域扩展）

    四层结构（重要：必须严格遵循）

    层 1
    wiki_type_index
    wiki 类型目录，如 database/、infra_cache/，按技术域分组。
    层 2
    类型 INDEX.md
    每个 wiki_type 有一个 INDEX.md，条目格式严格为：| 名称 | 描述 | 触发时机 | 文档链接 |
    层 3
    wiki_item.md
    完整问题描述 + 反例（失败案例）+ 正例（成功案例），指向原始日志。
    层 4
    原始日志
    完整的 .jsonl 对话记录或 .txt 导出，可完整溯源。
    wiki_tree/ ├── database/ │ ├── INDEX.md ← 层 2：| 名称 | 描述 | 触发时机 | 链接 | │ ├── postgres_pool.md ← 层 3：问题+反例+正例 │ └── postgres_memory.md ├── infra_cache/ │ ├── INDEX.md │ └── redis_lru_session_eviction.md └── infra_queue/ ├── INDEX.md └── celery_retry_storm.md
    INDEX.md 中的"触发时机"列是触发器的直接依据：haiku-agent 把用户消息和各 INDEX.md 的触发时机做语义对比，命中则热加载对应 item.md。
    🤖

    验证任务 5 · MiniMax 驱动的 Agentic 搜索

    wiki 搜索本身必须是 AI 驱动的，而非简单关键词匹配
    • 搜
      语义搜索：用户消息 → MiniMax M2.7 高速版 → 与 INDEX.md 各条目做语义相似度对比，返回得分最高的 top-k 条目。语义匹配
    • 验
      Agentic 验证：MiniMax agent 不仅做搜索，还做一次 JSON-in / JSON-out 的校验 + 改写，把原始 insight 适配到当前上下文。
    • 降
      优雅降级：即使 MiniMax API 调用失败，fallback 会把原卡的 fix 字段以 verdict=adopt 原样返回。demo 永远不会硬崩。fallback
    • 速
      速度目标：fast path（热加载 + adapter）约 1.3 秒；slow path 基线约 62 秒。差距约 48×。
    为什么必须是 Agentic 搜索？
    纯关键词匹配会错过语义相关但措辞不同的案例。MiniMax-driven search 能识别"checkout API timeout + postgres rejecting connections"和"postgres 连接池被打爆"是同一类问题，即使用词完全不同。
